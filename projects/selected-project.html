Galactic Cruiser

Embedded game system on the TI MSPM0G3507 LaunchPad.

Samuel Li and Travis Edwards

Class winner, ECE319H embedded handheld competition at UT Austin

Code Demo video

Galactic Cruiser is an embedded game project built on the MSPM0G3507 LaunchPad with a small TFT display, direct hardware inputs, and microcontroller-driven audio. The project was built for ECE319H at UT Austin and won the class embedded handheld competition.

The system uses the course's bare-metal style: peripheral initialization, timer-driven sampling, memory-constrained bitmap assets, and explicit driver boundaries for display, switches, analog input, I2C sensor reads, LEDs, and sound output.

Hardware Runtime

The runtime is designed around the MSPM0G3507 microcontroller and a 160 by 128 ST7735R display. The hardware layer initializes the system clock, LaunchPad GPIO, switch inputs, ADC-backed analog input, DAC-backed sound output, and I2C access for sensor reads.

A timer interrupt samples hardware state at a fixed cadence and stores input data for the main loop. The main loop owns display updates, which keeps ST7735 drawing serialized and avoids mixing slow screen writes directly into interrupt context.

Game Layer

The game layer sits above the driver code and treats the display as a constrained rendering target: sprites are preconverted into small bitmap assets, state changes are drawn explicitly, and user input is mapped from physical switches and analog readings into game actions.

That structure keeps the project close to the hardware. Instead of relying on an operating system, graphics engine, or dynamic asset loader, the game is built from static assets, deterministic update loops, and direct register-level peripheral code.

Audio And Feedback

The sound layer is organized around sampled audio assets and a DAC output path. Sound effects are represented as byte arrays, and the playback interface starts a selected sample while the interrupt side advances through the waveform.

LEDs and lightweight debug toggles provide visibility into timing and interrupt behavior during development. Those feedback paths matter on an embedded game because animation, input response, and audio timing all share a small real-time budget.
